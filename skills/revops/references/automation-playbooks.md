# 自动化工作流配方

针对 HubSpot、Salesforce、调度工具和跨工具自动化的平台特定工作流配方。

## HubSpot 工作流配方

### 1. MQL 警报和分配

**名称：** MQL 通知和任务创建
**触发器：** 联系人属性"生命周期阶段"更改为"营销合格潜在客户"
**操作：**
1. 在销售团队中轮询联系人所有者（轮询）
2. 向联系人所有者发送带有潜在客户上下文的内部电子邮件通知
3. 创建任务："跟进 [Contact Name]" — 4 小时内到期
4. 向 #sales-alerts 频道发送 Slack 通知
5. 注册到"MQL 跟进"序列（如果使用 HubSpot Sequences）
**结果：** 每个 MQL 都会立即分配并具有明确的 SLA
**注意：** 设置注册条件以排除已由代表拥有的潜在客户

---

### 2. MQL SLA 升级

**名称：** MQL SLA 违规警报
**触发器：** 联系人属性"生命周期阶段"等于"MQL"且"距离上次联系的天数"大于 0.5（12 小时）
**操作：**
1. 向联系人所有者发送内部电子邮件："SLA 警告：[Contact Name] 尚未联系"
2. 如果 24 小时后仍无活动 → 向销售经理发送警报
3. 如果 48 小时后仍无活动 → 通过轮询重新分配联系人所有者
4. 为新所有者创建任务："紧急：联系 [Contact Name] — 因 SLA 违规重新分配"
**结果：** 没有 MQL 会超过 48 小时无人处理
**注意：** 排除最后活动类型为"电话"或"会议"的联系人（已参与）

---

### 3. 潜在客户评分更新和 MQL 提升

**名称：** 达到分数阈值时自动 MQL
**触发器：** 联系人属性"HubSpot Score"大于或等于 65
**操作：**
1. 将生命周期阶段设置为"营销合格潜在客户"
2. 将"MQL 日期"设置为当前日期
3. 从营销培育工作流中抑制
4. 触发 MQL 警报工作流（配方 #1）
**结果：** 当潜在客户达到评分阈值时自动提升为 MQL
**注意：** 为现有客户和竞争对手添加抑制列表

---

### 4. 会议预订通知

**名称：** 会议预订警报给 AE
**触发器：** 记录联系人的会议活动（通过 Calendly/HubSpot 会议）
**操作：**
1. 向联系人所有者发送带有会议详细信息的内部电子邮件
2. 将联系人属性"上次会议预订"更新为当前日期
3. 如果生命周期阶段为"潜在客户" → 更新为"MQL"
4. 创建任务："准备与 [Contact Name] 的会议" — 会议前 1 小时到期
5. 向 #meetings 频道发送 Slack 通知
**结果：** AE 为每次会议做好充分准备
**注意：** 在通知电子邮件中包含最近的页面访问和内容下载

---

### 5. 成交后移交给 CS

**名称：** 客户入职触发器
**触发器：** 交易阶段更改为"成交"
**操作：**
1. 将关联联系人生命周期阶段更新为"客户"
2. 将"客户开始日期"设置为当前日期
3. 根据细分/区域将联系人所有者分配给 CS 团队成员
4. 为 CS 创建任务："安排与 [Company Name] 的启动电话" — 2 个工作日内到期
5. 将联系人注册到"客户入职"电子邮件序列
6. 向 CS 经理发送内部通知
7. 从所有销售序列中移除
**结果：** 从销售到客户服务的无缝交接
**注意：** 在 CS 通知中包含交易说明、合同价值和关键利益相关者

---

### 6. 陈旧交易警报

**名称：** 管道卫生 — 陈旧交易检测
**触发器：** 交易属性"当前阶段天数"大于 [该阶段平均值的 2 倍]
**操作：**
1. 向交易所有者发送内部电子邮件："交易陈旧警报：[Deal Name] 已在 [Stage] 停留 [X] 天"
2. 创建任务："更新或关闭 [Deal Name]" — 3 个工作日内到期
3. 如果 7 天后未更新 → 提醒销售经理
4. 添加到"陈旧交易"仪表板列表
**结果：** 管道保持清洁，预测保持准确
**注意：** 为每个阶段自定义阈值（发现：14 天，提案：10 天，谈判：21 天）

---

### 7. 回收的潜在客户培育重新进入

**名称：** MQL 回收到培育
**触发器：** 联系人属性"销售拒绝原因"已知（任何值）
**操作：**
1. 将生命周期阶段更新为"回收"
2. 将参与分数重置为基线（保留适配分数）
3. 注册到"回收的潜在客户培育"序列（较低频率）
4. 将"回收日期"设置为当前日期
5. 设置重新注册触发器：如果 HubSpot Score 再次超过阈值，重新触发 MQL 工作流
**结果：** 被拒绝的潜在客户获得第二次机会而不会堵塞管道
**注意：** 将回收到 MQL 的转化率作为单独的指标进行跟踪

---

### 8. 潜在客户活动摘要

**名称：** 每日潜在客户活动摘要
**触发器：** 计划 — 每天上午 8:00 本地时间
**操作：**
1. 过滤联系人：生命周期阶段为"SQL"或"机会"且在过去 24 小时内有网站活动
2. 向每个联系人所有者发送包含其潜在客户活动的摘要电子邮件
3. 包括：访问的页面、下载的内容、打开/点击的电子邮件
**结果：** 销售代表每天开始时都知道哪些潜在客户是活跃的
**注意：** 仅包含具有有意义活动的潜在客户（排除单次主页访问）

## Salesforce Flow 等效项

### 1. MQL 警报和分配（Salesforce Flow）

**类型：** 记录触发流
**对象：** Lead
**触发器：** Lead 字段"Status"更改为"MQL"
**流步骤：**
1. 获取记录：查询"Rep Assignment"自定义对象以获取下一个可用代表
2. 更新记录：将 Lead Owner 设置为分配的代表
3. 创建记录：创建任务 — "联系 MQL: {Lead.Name}"，截止日期 = NOW + 4 小时
4. 操作：向新的 lead owner 发送电子邮件警报
5. 更新记录：更新"Rep Assignment"上次分配的时间戳
**注意：** 使用自定义"Rep Assignment"对象来管理轮询状态

### 2. SLA 升级（Salesforce Flow）

**类型：** 计划触发流
**计划：** 工作时间内每 4 小时
**流步骤：**
1. 获取记录：Status = "MQL"且 LastActivityDate < TODAY - 1 的 Leads
2. 决策：lead 是否超过 48 小时无活动？
   - 是 → 重新分配给下一个代表，创建紧急任务，提醒经理
   - 否 → 向当前所有者发送提醒电子邮件
**注意：** 与 Process Builder 配对以在初始分配时进行实时警报

### 3. 管道阶段自动化（Salesforce Flow）

**类型：** 记录触发流
**对象：** Opportunity
**触发器：** Stage 字段已更新
**流步骤：**
1. 决策：它被更改为哪个阶段？
2. 对于每个阶段：
   - **发现：** 创建任务"完成发现问卷"
   - **演示：** 创建任务"准备演示环境"
   - **提案：** 创建任务"发送提案" + 如果 ACV > $25K 则提醒交易台
   - **成交：** 触发 CS 交接（创建 Case，分配 CS 所有者，发送欢迎电子邮件）
   - **流失：** 创建任务"记录流失原因" + 添加到赢/输分析报告

### 4. 陈旧交易检测（Salesforce Flow）

**类型：** 计划触发流
**计划：** 每天上午 7:00
**流步骤：**
1. 获取记录：Days_In_Stage > Stage_SLA_Threshold 的开放 Opportunities
2. 循环遍历结果：
   - 创建任务："更新陈旧交易：{Opportunity.Name}"
   - 向 Opportunity Owner 发送电子邮件
   - 如果 Days_In_Stage > 2x 阈值 → 向 Owner 的 Manager 发送电子邮件
3. 更新自定义字段"Stale Flag" = true 以实现仪表板可见性

## Calendly / SavvyCal 集成模式

### 轮询会议调度

**Calendly 设置：**
1. 创建包含所有合格代表的团队事件类型
2. 分配："优化平等分配"
3. 可用性：每个代表管理自己的日历
4. 缓冲：会议前后 15 分钟
5. 最短通知：4 小时（避免最后一刻预订）

**CRM 集成：**
1. Calendly webhook 在预订时触发
2. 将邀请者电子邮件与 CRM 联系人匹配
3. 如果联系人存在 → 将会议分配给联系人所有者（如果拥有则覆盖轮询）
4. 如果新联系人 → 创建 lead，通过路由规则分配，记录会议
5. 将生命周期阶段设置为 MQL（会议 = 高意向）

### SavvyCal 设置

**相对于 Calendly 的优势：**
- 基于优先级的调度（偏好某些时间段）
- 叠加日历（在一个视图中显示团队可用性）
- 每个代表的个性化预订链接

**集成模式：**
1. 创建具有优先级规则的团队调度链接
2. 预订时的 webhook → Zapier/Make → CRM
3. 匹配或创建联系人，分配所有者，创建任务
4. 发送包含会议准备材料的确认

### 按标准路由会议

```
提交预订表单
├─ 公司规模 > 500？（表单字段）
│  ├─ 是 → 路由到企业 AE 日历
│  └─ 否 ↓
├─ 现有客户？（CRM 查找）
│  ├─ 是 → 路由到账户所有者的日历
│  └─ 否 ↓
└─ 在 SDR 团队之间轮询
```

### 缺席工作流

**触发器：** 会议时间过去 + 30 分钟内未记录会议笔记
**操作：**
1. 等待预定会议时间后 30 分钟
2. 检查：是否记录了电话或会议？
   - 是 → 无操作
   - 否 → 向潜在客户发送"抱歉我们错过了你"电子邮件
3. 创建任务："与 [Contact Name] 重新安排" — 下一个工作日到期
4. 如果第二次缺席 → 标记联系人并提醒经理

## Zapier 跨工具模式

### 1. 新潜在客户 → CRM + Slack + 任务

**触发器：** 新表单提交（Typeform、HubSpot、Webflow）
**操作：**
1. 在 CRM 中创建/更新联系人
2. 使用 Clearbit 丰富（如果可用）
3. 在 Slack #new-leads 中发布丰富数据
4. 在项目管理工具（Asana、Linear）中创建任务

### 2. 会议预订 → CRM + 准备电子邮件

**触发器：** 新 Calendly/SavvyCal 预订
**操作：**
1. 查找或创建 CRM 联系人
2. 将生命周期阶段更新为 MQL
3. 向分配的代表发送准备电子邮件（包括 CRM 链接、LinkedIn 个人资料、最近活动）
4. 创建会前任务

### 3. 交易成交 → 入职堆栈

**触发器：** CRM 交易阶段更改为"成交"
**操作：**
1. 在 CS 工具（Vitally、Gainsight、ChurnZero）中创建客户记录
2. 添加到入职项目模板
3. 通过电子邮件工具发送欢迎电子邮件
4. 创建 Slack 频道：#customer-[company-name]
5. 在 Slack 中通知 CS 团队

### 4. 潜在客户评分 → 跨工具同步

**触发器：** CRM 潜在客户分数跨越 MQL 阈值
**操作：**
1. 更新营销自动化平台状态
2. 添加到再营销受众（Facebook、Google Ads）
3. 触发 SDR 外展序列
4. 在分析工具中记录事件（Mixpanel、Amplitude）

### 5. SLA 违规 → 多渠道警报

**触发器：** CRM 任务逾期（MQL 跟进任务）
**操作：**
1. 向代表发送 Slack DM
2. 向代表发送电子邮件
3. 如果逾期 2+ 小时 → 向经理发送 Slack DM
4. 如果逾期 4+ 小时 → 在 CRM 中重新分配（通过 webhook 返回 CRM）

### 6. 每周管道摘要

**触发器：** 计划 — 每周一上午 8:00
**操作：**
1. 查询 CRM 以获取管道摘要（总价值、新交易、陈旧交易、预期关闭）
2. 格式化为摘要
3. 发布到 Slack #sales-team
4. 向销售领导层发送电子邮件摘要
