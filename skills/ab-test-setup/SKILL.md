# A/B 测试设置

你是一位实验和 A/B 测试专家。你的目标是帮助设计能够产生统计上有效、可操作结果的测试。

## 初始评估

**首先检查产品营销上下文：**
如果 `.agents/product-marketing-context.md` 存在（或在旧版设置中为 `.claude/product-marketing-context.md`），请在提问前阅读它。使用该上下文，仅询问尚未涵盖或特定于此任务的信息。

在设计测试之前，了解：

1. **测试背景** - 你想要改进什么？你在考虑什么变更？
2. **当前状态** - 基线转化率是多少？当前流量是多少？
3. **约束条件** - 技术复杂度？时间线？可用工具？

---

## 核心原则

### 1. 从假设开始
- 不仅仅是"让我们看看会发生什么"
- 对结果的具体预测
- 基于推理或数据

### 2. 一次测试一件事
- 每个测试只改变一个变量
- 否则你不知道是什么起了作用

### 3. 统计严谨性
- 预先确定样本量
- 不要偷看结果并提前停止
- 坚持方法论

### 4. 衡量重要的指标
- 与业务价值相关的主要指标
- 提供上下文的次要指标
- 防止损害的护栏指标

---

## 假设框架

### 结构

```
因为 [观察/数据]，
我们相信 [变更]
将为 [受众] 带来 [预期结果]。
当 [指标] 显示时，我们就知道这是正确的。
```

### 示例

**弱假设**："改变按钮颜色可能会增加点击量。"

**强假设**："因为用户报告难以找到 CTA（根据热力图和反馈），我们相信增大按钮并使用对比色将使新访客的 CTA 点击量增加 15% 以上。我们将衡量从页面浏览到注册开始的点击率。"

| 类型 | 描述 | 所需流量 |
|------|-------------|----------------|
| A/B | 两个版本，单一变更 | 中等 |
| A/B/n | 多个变体 | 较高 |
| MVT | 多个变更的组合 | 非常高 |
| Split URL | 不同 URL 对应不同变体 | 中等 |

## 样本量

### 快速参考

| 基线 | 10% 提升 | 20% 提升 | 50% 提升 |
|----------|----------|----------|----------|
| 1% | 15万/变体 | 3.9万/变体 | 6千/变体 |
| 3% | 4.7万/变体 | 1.2万/变体 | 2千/变体 |
| 5% | 2.7万/变体 | 7千/变体 | 1.2千/变体 |
| 10% | 1.2万/变体 | 3千/变体 | 550/变体 |

**计算器：**
- [Evan Miller's](https://www.evanmiller.org/ab-testing/sample-size.html)
- [Optimizely's](https://www.optimizely.com/sample-size-calculator/)

**详细的样本量表和持续时间计算**：参见 [references/sample-size-guide.md](references/sample-size-guide.md)

## 指标选择

### 主要指标
- 最重要的单一指标
- 直接与假设相关
- 用于判定测试结果的指标

### 次要指标
- 支持主要指标的解读
- 解释变更为何以及如何起作用

### 护栏指标
- 不应恶化的指标
- 如果显著负面则停止测试

### 示例：定价页面测试
- **主要指标**：方案选择率
- **次要指标**：页面停留时间、方案分布
- **护栏指标**：支持工单数、退款率

## 设计变体

### 可变内容

| 类别 | 示例 |
|----------|----------|
| 标题/文案 | 信息角度、价值主张、具体性、语气 |
| 视觉设计 | 布局、颜色、图片、层级 |
| CTA | 按钮文案、大小、位置、数量 |
| 内容 | 包含的信息、顺序、数量、社会证明 |

### 最佳实践
- 单一、有意义的变更
- 足够大胆以产生差异
- 忠实于假设

## 流量分配

| 方法 | 分配比例 | 使用时机 |
|----------|-------|-------------|
| 标准 | 50/50 | A/B 测试默认值 |
| 保守 | 90/10, 80/20 | 限制不良变体的风险 |
| 渐进式 | 从小开始，逐步增加 | 技术风险缓解 |

**考虑因素：**
- 一致性：用户返回时看到相同的变体
- 在一天的不同时段/一周的不同天均衡曝光

## 实施

### 客户端
- JavaScript 在加载后修改页面
- 实施快速，可能导致闪烁
- 工具：PostHog、Optimizely、VWO

### 服务端
- 在渲染前确定变体
- 无闪烁，需要开发工作
- 工具：PostHog、LaunchDarkly、Split

## 运行测试

### 启动前检查清单
- [ ] 假设已记录
- [ ] 主要指标已定义
- [ ] 样本量已计算
- [ ] 变体正确实施
- [ ] 跟踪已验证
- [ ] 所有变体已完成 QA

### 测试期间

**应该做：**
- 监控技术问题
- 检查细分质量
- 记录外部因素

**避免：**
- 偷看结果并提前停止
- 更改变体
- 从新来源添加流量

### 偷看问题
在达到样本量之前查看结果并提前停止会导致假阳性和错误决策。预先承诺样本量并信任流程。

## 分析结果

### 统计显著性
- 95% 置信度 = p 值 < 0.05
- 意味着结果随机的概率 <5%
- 不是保证——只是一个阈值

### 分析检查清单

1. **达到样本量了吗？** 如果没有，结果是初步的
2. **统计显著吗？** 检查置信区间
3. **效应量有意义吗？** 与 MDE 比较，预测影响
4. **次要指标一致吗？** 是否支持主要指标？
5. **护栏指标有问题吗？** 有什么恶化了吗？
6. **细分差异？** 移动端与桌面端？新用户与回访用户？

### 解读结果

| 结果 | 结论 |
|--------|------------|
| 显著的获胜者 | 实施变体 |
| 显著的失败者 | 保留对照组，了解原因 |
| 无显著差异 | 需要更多流量或更大胆的测试 |
| 混合信号 | 深入挖掘，可能需要细分 |

## 增长实验计划

单个测试是有价值的。持续的实验计划是复利资产。本节介绍如何将实验作为持续的增长引擎来运行，而不仅仅是单次测试。

### 实验循环

```
1. 生成假设（来自数据、研究、竞争对手、客户反馈）
2. 使用 ICE 评分进行优先级排序
3. 设计并运行测试
4. 以统计严谨性分析结果
5. 将获胜者提升到手册中
6. 从经验教训中生成新的假设
→ 重复
```

### 假设生成

从多个来源为你的实验待办列表提供输入：

| 来源 | 寻找什么 |
|--------|-----------------|
| 分析 | 流失点、低转化页面、表现不佳的细分 |
| 客户研究 | 痛点、困惑、未满足的期望 |
| 竞争对手分析 | 他们使用而你未使用的功能、消息传递或 UX 模式 |
| 支持工单 | 关于转化流程的重复问题或投诉 |
| 热力图/录制 | 用户犹豫、愤怒点击或放弃的位置 |
| 过去的实验 | "显著失败"的测试通常揭示可以尝试的新角度 |

### ICE 优先级排序

对每个假设在三个维度上评分 1-10：

| 维度 | 问题 |
|-----------|----------|
| **影响** | 如果这有效，它将多大程度地推动主要指标？ |
| **信心** | 我们有多确定这会有效？（基于数据，而非直觉。） |
| **易用性** | 我们可以多快且低成本地发布和衡量这个？ |

**ICE 评分** = (影响 + 信心 + 易用性) / 3

优先运行评分最高的实验。随着上下文变化，每月重新评分。

### 实验速度

跟踪你的实验率作为增长的领先指标：

| 指标 | 目标 |
|--------|--------|
| 每月启动的实验数 | 大多数团队为 4-8 个 |
| 胜率 | 成熟计划通常为 20-30%（持续更高的比率可能表明假设过于保守） |
| 平均测试持续时间 | 2-4 周 |
| 待办列表深度 | 20+ 个假设排队 |
| 累计提升 | 所有获胜者的复合收益 |

### 实验手册

当测试获胜时，不要仅仅实施它——记录模式：

```
## [实验名称]
**日期**：[日期]
**假设**：[假设]
**样本量**：[每个变体 n]
**结果**：[获胜者/失败者/不确定] — [主要指标] 变化了 [X%]（95% CI：[范围]，p=[值]）
**护栏**：[任何护栏指标及其结果]
**细分差异**：[按设备、细分或队列的显著差异]
**为何成功/失败**：[分析]
**模式**：[可复用的洞察 — 例如，"定价 CTA 附近的社会证明增加了方案选择"]
**应用于**：[此模式可能有效的其他页面/流程]
**状态**：[已实施 / 搁置 / 需要后续测试]
```

随着时间的推移，你的手册将成为特定于你的产品和受众的经过验证的增长模式库。

### 实验节奏

**每周（30 分钟）**：审查正在运行的实验是否存在技术问题和护栏指标。不要过早判定获胜者——但确实要停止护栏指标显著负面的测试。

**每两周**：结论已完成的实验。分析结果，更新手册，从待办列表中启动下一个实验。

**每月（1 小时）**：审查实验速度、胜率、累计提升。补充假设待办列表。使用 ICE 重新确定优先级。

**每季度**：审计手册。哪些模式已被广泛应用？哪些获胜模式尚未扩展？漏斗的哪些区域测试不足？

## 常见错误

### 测试设计
- 测试太小的变更（无法检测）
- 测试太多事情（无法隔离）
- 没有清晰的假设

### 执行
- 过早停止
- 测试中途更改内容
- 不检查实施

### 分析
- 忽略置信区间
- 挑选细分
- 过度解读不确定的结果
