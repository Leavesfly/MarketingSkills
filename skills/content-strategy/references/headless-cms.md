# 无头 CMS 指南

为营销内容选择、建模和实施无头 CMS 的参考。

## 何时使用此参考

当为新项目选择 CMS、为营销网站设计内容模型、设置编辑工作流程，或将 CMS 内容连接到程序化页面时使用此参考。

---

## 无头 CMS vs 传统 CMS

无头 CMS 将内容管理与展示分离。内容存储在结构化的后端，并通过 API 交付到任何前端。

### 何时适合使用无头 CMS

- 多个前端消费相同的内容（Web、移动应用、电子邮件）
- 开发人员希望完全控制前端技术栈
- 内容需要在多个渠道中重用
- 你正在使用现代框架构建（Next.js、Remix、Astro）
- 营销需要结构化、可重用的内容块

### 何时传统 CMS 更合适

- 小团队且没有专职开发人员
- 简单的博客或宣传网站
- WYSIWYG 编辑是硬性要求
- 预算紧张且 WordPress/Webflow 能胜任工作

### 决策清单

| 因素 | 无头 CMS | 传统 CMS |
|--------|----------|-------------|
| 多渠道交付 | 是 | 有限 |
| 开发人员控制权 | 完全 | 受限 |
| 非技术人员编辑 | 需要设置 | 内置 |
| 上线时间 | 较长 | 较快 |
| 内容重用 | 原生支持 | 手动 |
| 托管灵活性 | 任意前端 | 依赖平台 |

## 营销内容建模

### 核心原则

1. **以类型思考，而非页面。**"落地页"是一个带有字段的内容类型——而不是 HTML 文件。这让你可以在页面之间重用组件。
2. **将内容与展示分离。**存储标题文本，而不是样式化的标题。展示属于前端。
3. **为重用而设计。**如果推荐出现在 5 个页面上，创建 Testimonial 类型并引用它——不要复制。
4. **保持模型扁平。**深层嵌套结构难以查询和维护。优先使用引用而非嵌套。

### 常见营销内容类型

| 类型 | 关键字段 | 说明 |
|------|-----------|-------|
| **落地页** | title, slug, hero, sections[], seo | 模块化部分以实现灵活性 |
| **博客文章** | title, slug, body, author, category, tags, publishedAt, seo | 富文本或 Portable Text 正文 |
| **案例研究** | title, customer, challenge, solution, results, metrics[], logo | 链接到相关产品/功能 |
| **推荐语** | quote, author, role, company, avatar, rating | 从落地页引用 |
| **FAQ** | question, answer, category | 按类别分组用于程序化页面 |
| **作者** | name, bio, avatar, social links | 从博客文章引用 |
| **CTA 块** | heading, body, buttonText, buttonUrl, variant | 在页面间重用 |

### SEO 字段清单

每个页面级内容类型都需要：

- `metaTitle` — 50-60 个字符
- `metaDescription` — 150-160 个字符
- `ogImage` — 1200x630px 社交预览图
- `slug` — URL 路径段
- `canonicalUrl` — 可选覆盖
- `noIndex` — 布尔值，用于从搜索中排除
- `structuredData` — 可选 JSON-LD 覆盖

## 编辑工作流程

### 草稿 → 审核 → 发布周期

1. **草稿** — 作者创建或编辑内容
2. **审核** — 编辑审核准确性、品牌声音、SEO
3. **批准** — 利益相关者签字确认
4. **调度** — 设置发布日期/时间
5. **发布** — 内容通过 API 上线

### 预览 API

所有主要的无头 CMS 平台都支持草稿预览：

- **Sanity**: 使用 `useLiveQuery` 或 Presentation 工具进行实时预览
- **Contentful**: 预览 API（`preview.contentful.com`）带有单独的访问令牌
- **Strapi**: 草稿和发布系统，带有 `status=draft` 查询参数（v5；取代 v4 的 `publicationState`）

在前端设置预览路由（例如 `/api/preview`），进行身份验证并渲染草稿内容。

### 角色和权限

| 角色 | 可以创建 | 可以编辑 | 可以发布 | 可以删除 |
|------|:----------:|:--------:|:-----------:|:----------:|
| 作者 | 是 | 自己的 | 否 | 自己的草稿 |
| 编辑 | 是 | 全部 | 是 | 草稿 |
| 管理员 | 是 | 全部 | 是 | 全部 |

确切的权限模型因平台而异。Sanity 使用基于角色的访问。Contentful 有空间级角色。Strapi 有细粒度的 RBAC。

## 平台比较

| 功能 | Sanity | Contentful | Strapi |
|---------|--------|------------|--------|
| 托管 | 云（托管） | 云（托管） | 自托管或云 |
| 查询语言 | GROQ | REST / GraphQL | REST / GraphQL |
| 免费层级 | 慷慨 | 有限 | 开源（免费） |
| 实时协作 | 是（内置） | 有限 | 否 |
| 最适合 | 开发人员灵活性 | 企业多语言环境 | 预算/自托管 |
| 内容建模 | 代码即架构 | Web UI | Web UI 或代码 |
| 媒体处理 | 内置 DAM | 内置 | 基于插件 |

### Sanity

**优势**: GROQ 查询语言强大且灵活。架构在代码中定义（版本控制）。实时协作编辑。Portable Text 用于丰富内容。慷慨的免费层级。

**注意事项**: 非开发人员的陡峭学习曲线。Studio 自定义需要 React 知识。GROQ 查询的供应商锁定。

**营销适配**: 最适合开发人员和营销人员密切协作时。对于具有复杂模型的内容密集型网站非常强大。

### Contentful

**优势**: 成熟的企业平台。出色的多语言支持。强大的集成生态系统。通过 Studio 实现可组合内容。文档完善的 API。

**注意事项**: 定价随内容类型和语言环境扩展。两个独立的 API（Delivery 和 Management）。较低层级的速率限制可能较紧。

**营销适配**: 最适合有多市场内容需求的企业。当你需要成熟的供应商可靠性时很好。

### Strapi

**优势**: 开源、自托管选项。完全控制数据。没有按席位定价。可自定义的管理面板。插件生态系统。默认 REST，通过插件支持 GraphQL。

**注意事项**: 自托管意味着你需要处理基础设施。生态系统比 Sanity/Contentful 小。从 V4 迁移到 V5 可能有重大变化。

**营销适配**: 最适合具有 DevOps 能力并希望完全控制且无供应商锁定的团队。适合预算敏感的项目。

### 其他值得了解的平台

- **Hygraph** — GraphQL 原生，擅长联合和多源内容
- **Keystatic** — 基于 Git，适合开发人员-内容混合工作流程
- **Payload** — TypeScript 优先，自托管，像 Sanity 一样通过代码配置
- **Builder.io** — 带有无头后端的可视化编辑器，适合非技术营销人员
- **Prismic** — 基于切片的内容建模，强大的 Next.js 集成

## 与营销技能的集成

### 程序化 SEO

使用 CMS 作为程序化页面的数据源。将结构化数据（FAQ、比较、城市页面）存储为内容类型，并从查询生成页面。参见 **programmatic-seo** 技能。

### 文案写作

CMS 内容模型强制执行一致的结构。定义与你的文案框架匹配的字段（标题、副标题、社会证明、CTA）。参见 **copywriting** 技能。

### 网站架构

URL 结构、导航层级和内部链接都取决于内容在 CMS 中的组织方式。一起规划你的内容模型和网站架构。参见 **site-architecture** 技能。

### 电子邮件序列

将 CMS 内容拉入电子邮件模板，以保持 Web 和电子邮件之间的一致性。案例研究、推荐语和博客文章可以为电子邮件培育序列提供内容。参见 **email-sequence** 技能。

## 实施清单

- [ ] 基于页面类型和可重用块定义内容类型
- [ ] 为每个页面级内容类型添加 SEO 字段
- [ ] 在前端设置预览/草稿模式
- [ ] 为你的团队配置角色和权限
- [ ] 在构建前端之前为每种类型创建示例内容
- [ ] 设置内容变更的 webhook 通知（重建触发器）
- [ ] 为编辑记录内容指南（字段描述、字符限制）
- [ ] 测试内容交付性能（CDN、缓存、ISR）
- [ ] 如果从现有 CMS 迁移，规划迁移策略
